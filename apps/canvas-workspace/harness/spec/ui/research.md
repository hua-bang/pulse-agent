# UI consistency and component reuse research

Date: 2026-07-04. Scope: official documentation and first-party sources only.

This note studies Atlassian Design System, GitHub Primer, Shopify Polaris, Material Design / Material Web, and Storybook for practices that can inform `canvas-workspace` UI consistency without importing a heavyweight design-system program.

## Source Mechanisms

### Atlassian Design System

- Atlassian frames its system as foundations, components, patterns, and tools for shipping Atlassian interfaces; foundations include tokens and visual styles, components are reusable interaction building blocks, and patterns combine components and foundations into consistent experiences: https://atlassian.design/design-system
- Atlassian design tokens are the single source of truth for small repeatable design decisions such as color, typography, spacing, radius, border width, elevation, interaction states, and opacity: https://atlassian.design/llms-tokens.txt
- Atlassian recommends semantic token use over matching raw colors exactly, warns against hardcoded colors because they break dark mode, and calls out hover, focus, active, and disabled feedback as explicit tokenized states: https://atlassian.design/llms-tokens.txt
- Atlassian's component library is built on tokens and primitives to keep React components consistent, accessible, and responsive across Atlassian apps: https://atlassian.design/llms-components.txt
- Atlassian exposes enforcement and migration mechanisms, including design-system ESLint rules, a UI styling standard plugin, codemods, Figma libraries, and an official ADS MCP server: https://atlassian.design/llms.txt

### GitHub Primer

- Primer Product UI is described as guidelines, principles, and patterns for designing and building UI at GitHub, with areas for getting started, components, primitives, patterns, Octicons, and contribution guidance: https://primer.style/product/
- Primer primitives are distributed as design tokens for color, spacing, and typography; the package exposes tokens as CSS variables, including theme-specific color files for light, dark, dimmed, high-contrast, and colorblind modes: https://primer.style/product/primitives/
- Primer separates color tokens into base, functional, and component/pattern categories; base tokens map to raw values and should not be used directly in code or design, while functional and component/pattern tokens respect color modes: https://primer.style/product/getting-started/foundations/color-usage/
- Primer's Figma guidance keeps design and implementation aligned: the Primer Web Figma library contains UI components and tokens that match Primer React Components, Primer ViewComponents, and Primer CSS: https://primer.style/product/getting-started/figma/
- Primer requires new components to first exist with a name, description, at least one Storybook story, basic accessibility checks, design quality checks, documentation, and an accessibility path before upstreaming: https://primer.style/product/contribute/adding-new-components/
- Primer discourages duplicate components, asks teams to modify an existing Primer component first when possible, and upstreams only when multiple feature teams use the pattern and the maintainers judge the maintenance cost worthwhile: https://primer.style/product/contribute/adding-new-components/
- Primer explicitly allows app-local one-off patterns when they are accessible, visually consistent, and high quality; shareable candidates should be likely to serve more than one product, avoid app-specific side effects/dependencies, and replace multiple near-duplicate implementations: https://primer.style/product/contribute/handling-new-patterns/
- Primer component status distinguishes experimental, ready, and deprecated; ready components are mature enough for use and expected to have long-term support, while deprecated components should be avoided and documented with alternatives: https://primer.style/product/getting-started/component-status/
- Primer React's philosophy favors presentational components that standardize common patterns and avoid fetching/submitting data; consumers can wrap Primer components for data behavior: https://primer.style/product/getting-started/react/philosophy/
- Primer React classifies reusable UI as building blocks, pattern components, and helper components, and keeps controlled flexibility through mechanisms such as `className`, polymorphic `as`, slots, and wrappers: https://primer.style/product/getting-started/react/philosophy/ and https://primer.style/product/getting-started/react/extending-components/
- Primer React offers an ESLint plugin for best-practice enforcement, but the plugin is marked experimental in Primer's own docs: https://primer.style/product/getting-started/react/linting/

### Shopify Polaris

- Polaris React is now marked deprecated, while Shopify says Polaris Web Components are released and are now the default way to build Shopify apps: https://polaris-react.shopify.com/ and https://shopify.dev/docs/api/app-home/web-components
- Polaris Web Components provide UI building blocks for displaying data, capturing merchant input, and triggering API calls; Shopify states these components follow Shopify's design system so apps look and feel native to Shopify admin: https://shopify.dev/docs/api/app-home/web-components
- Polaris Web Components are framework-flexible because they are loaded as web components and can be added through a script tag; Shopify also provides companion TypeScript types through `@shopify/polaris-types`: https://shopify.dev/docs/api/app-home/web-components
- Shopify groups available Polaris web components by jobs such as actions, feedback/status, forms, layout/structure, overlays, media/visuals, and typography/content: https://shopify.dev/docs/api/app-home/web-components
- Shopify documents accessibility as built into Polaris web components through semantic HTML, keyboard navigation, ARIA attributes, focus management, color contrast, and warnings for missing required accessibility properties: https://shopify.dev/docs/api/polaris/using-polaris-web-components
- Polaris React still provides useful taxonomy: components are reusable building blocks for Shopify admin experiences, and deprecated components are explicitly listed alongside active categories: https://polaris-react.shopify.com/components
- Polaris patterns document preferred solutions to common merchant goals such as app settings layout, card layout, common actions, date picking, and resource index/detail layouts: https://polaris-react.shopify.com/patterns

### Material Design and Material Web

- Material Design describes itself as an adaptable system of guidelines, components, and tools supporting UI design best practices and backed by open-source code: https://m3.material.io/get-started
- Material Design states that design tokens are the building blocks of all UI elements and that the same tokens are used in designs, tools, and code: https://m3.material.io/foundations/design-tokens
- Material Web is Google's first-party web component library that follows Material Design guidelines; its components are custom elements with encapsulated styles and behavior that work across frameworks such as Lit, React, Vue, and Svelte: https://material-web.dev/about/intro/
- Material Web treats tokens as CSS custom properties; reference tokens hold concrete values, system tokens define roles for color, typography, elevation, and shape, and component tokens map component attributes to system tokens or concrete values: https://material-web.dev/theming/material-theming/
- Material Web component docs combine usage, accessibility, theming tokens, API documentation, and source links; the button docs include five button types, label guidance, ARIA guidance, focusability behavior, and token tables: https://material-web.dev/components/button/
- Material Web explicitly notes current token gaps: MWC does not currently support `--md-ref-palette` tokens or `--md-sys-motion` tokens: https://material-web.dev/theming/material-theming/

### Storybook

- Storybook describes itself as a frontend workshop for building UI components and pages in isolation; stories track component and page use cases and can document UI for team reuse: https://storybook.js.org/docs/get-started/why-storybook
- Storybook says stories make UI states reusable as documentation, interaction tests, accessibility tests, visual tests, and test cases imported into other JavaScript testing tools: https://storybook.js.org/docs/get-started/why-storybook and https://storybook.js.org/docs/writing-tests
- Storybook Controls provide a graphical UI for editing component args in real time; controls are generated from story args and make stories portable across documentation, tests, and design review: https://storybook.js.org/docs/essentials/controls
- Storybook recommends pure presentational pages as one page-story strategy because all story data can live in args and connected app logic can stay in an outer wrapper: https://storybook.js.org/docs/writing-stories/build-pages-with-storybook
- Storybook parameters provide static metadata at project, component, and story levels; they merge by specificity and can standardize layout or addon behavior across stories: https://storybook.js.org/docs/api/parameters
- Storybook tags can include or exclude stories for different workflows such as development, docs, and test runs; custom tags can encode status, persona, or ownership: https://storybook.js.org/docs/writing-stories/tags
- Storybook supports visual testing by turning each story into a visual test when enabled, but its native cross-browser visual testing path depends on Chromatic, a cloud service made by the Storybook team: https://storybook.js.org/docs/writing-tests
- Storybook design integrations can embed Storybook in Figma or Figma in Storybook, but the Figma plugin path requires a Storybook published to Chromatic: https://storybook.js.org/docs/sharing/design-integrations

## Transferable Principles For `canvas-workspace`

- Use a local semantic-token contract before adding more components. `canvas-workspace` already has CSS custom properties for layers and visual style; the next step should be a small documented set of semantic tokens for surface, border, text, focus, radius, spacing, and state rather than raw colors in component CSS. This follows Atlassian's single-source token approach and Material Web's CSS-custom-property token model: https://atlassian.design/llms-tokens.txt and https://material-web.dev/theming/material-theming/
- Prefer semantic tokens over raw visual matching. Atlassian says to prefer semantically correct tokens even when the color does not match exactly, and Primer says base tokens should not be used directly because functional/component tokens carry color-mode behavior. For this repo, that means a new canvas panel should choose `--ui-surface-raised` or `--ui-border-focus` style roles instead of copying a nearby OKLCH literal: https://atlassian.design/llms-tokens.txt and https://primer.style/product/getting-started/foundations/color-usage/
- Keep the reuse ladder explicit: primitive, component, pattern, surface. Atlassian separates foundations, components, and patterns, and Primer separates building blocks, pattern components, and helper components. For `canvas-workspace`, local primitives can cover layout/text/button/input behaviors, while patterns should cover repeated workbench tasks such as tabbed previews, command rows, empty states, progress/loading, and form settings: https://atlassian.design/design-system and https://primer.style/product/getting-started/react/philosophy/
- Promote only repeated, side-effect-free UI into shared components. Primer's upstreaming guidance asks whether a pattern is used by multiple teams, avoids app-specific side effects/dependencies, and replaces duplicate near-patterns. For `canvas-workspace`, a chat-only widget or one-off canvas node body can remain local until at least two surfaces need the same behavior and API: https://primer.style/product/contribute/handling-new-patterns/ and https://primer.style/product/contribute/adding-new-components/
- Treat one-off UI as acceptable when it is accessible, visually consistent, and high quality. Primer explicitly allows app-local one-off patterns under those constraints, which fits a product like Pulse Canvas where some node/body interactions may be highly specialized: https://primer.style/product/contribute/handling-new-patterns/
- Require a small component readiness checklist before declaring shared UI. A candidate shared component should have a name, purpose, owner/consumer surfaces, supported states, accessibility notes, and at least one runnable example or harness scenario. This is the lightweight version of Primer's requirement for a name, description, Storybook story, design checks, documentation, and accessibility path: https://primer.style/product/contribute/adding-new-components/
- Keep shared components presentational; place data loading, IPC, and side effects in wrappers/hooks. Primer React favors presentational components and Storybook says presentational pages make story data portable through args. This matches `canvas-workspace` renderer conventions, where heavy state/effect logic should move into hooks and privileged behavior goes through `window.canvasWorkspace`: https://primer.style/product/getting-started/react/philosophy/ and https://storybook.js.org/docs/writing-stories/build-pages-with-storybook
- Document component states as examples before building a full design-system site. Storybook's core mechanism is reusable stories, and Primer uses Storybook as an early discoverability criterion. In this repo, the first low-cost equivalent can be colocated example fixtures or harness screenshots for key states such as loading, empty, error, selected, focused, collapsed, and disabled: https://storybook.js.org/docs/get-started/why-storybook and https://primer.style/product/contribute/adding-new-components/
- Make accessibility part of the component API, not a late audit. Shopify documents built-in labels, keyboard navigation, ARIA, focus management, and contrast as component responsibilities; Material Web component docs include ARIA and focusability guidance. For `canvas-workspace`, reusable inputs, icon buttons, tabs, menus, dialogs, and dock controls should expose labels, focus behavior, keyboard affordances, and disabled semantics in their props/examples: https://shopify.dev/docs/api/polaris/using-polaris-web-components and https://material-web.dev/components/button/
- Use status labels for reusable UI. Primer's experimental/ready/deprecated states and Polaris' explicit deprecated component list show that reuse needs lifecycle language. A local `experimental`, `shared`, and `deprecated` label in docs or examples would be enough until there is an automated catalog: https://primer.style/product/getting-started/component-status/ and https://polaris-react.shopify.com/components
- Keep escape hatches narrow and named. Primer's extension mechanisms separate polymorphic rendering, slots, and wrappers; this is a better model than ad hoc prop expansion. For `canvas-workspace`, use typed slots/composition for specific placement needs and wrappers for behavior defaults, instead of adding arbitrary styling props to every component: https://primer.style/product/getting-started/react/extending-components/
- Prefer current repo harness before a new toolchain. Storybook is valuable when stories can become the source of truth for docs/tests, but the repo already has an Electron runtime harness for real app surfaces. For now, use local harness screenshots and focused examples to prove repeated states before introducing another always-on UI catalog: https://storybook.js.org/docs/writing-tests and https://storybook.js.org/docs/get-started/why-storybook

## Heavy Tools Not Recommended Now

- Do not import a full external component library as the canvas UI foundation right now. Atlassian, Primer, Shopify, and Material all encode product-specific interaction language and theming assumptions; Material Web also uses encapsulated web components, and Polaris Web Components target Shopify admin-native app surfaces. Borrow the mechanisms, not the package dependency: https://material-web.dev/about/intro/ and https://shopify.dev/docs/api/app-home/web-components
- Do not introduce Storybook plus Chromatic as a required gate now. Storybook's visual-testing path turns stories into tests, but native cross-browser visual testing depends on Chromatic, and the Figma integration path also requires a Storybook published to Chromatic. This repo currently has no CI or mechanical harness gates, so adding a cloud visual-review dependency before a stable story corpus would create process weight ahead of value: https://storybook.js.org/docs/writing-tests and https://storybook.js.org/docs/sharing/design-integrations
- Do not add a Figma-library or token-sync pipeline now. Primer's Figma libraries work because GitHub treats Figma as a design-team source of truth with libraries matching developer components, but this repo does not currently have that external design source of truth. Start with code-owned semantic CSS tokens and add design-tool sync only after token names and component states settle: https://primer.style/product/getting-started/figma/
- Do not add design-system ESLint plugins or codemod migrations as a first step. Atlassian's enforcement stack includes ESLint plugins, styling standards, codemods, and MCP tooling, and Primer's React ESLint plugin is itself marked experimental. Because this repo currently has no lint/format stack, a new lint regime should follow a stable local rule set, not precede it: https://atlassian.design/llms.txt and https://primer.style/product/getting-started/react/linting/
- Do not replace colocated CSS with CSS-in-JS or external styling conventions for this effort. Atlassian's packages and styling standards are built around its own stack, while `canvas-workspace` already documents colocated `index.css` and existing token usage. The near-term consistency win is token naming and shared component APIs, not changing the styling substrate: https://atlassian.design/llms-styling.txt
- Do not build a public design-system documentation site yet. Primer and Polaris documentation reflect mature systems with contribution, status, deprecation, design, and implementation workflows. `canvas-workspace` first needs a small internal contract for tokens, readiness, examples, and repeated patterns before a standalone docs surface would pay for its maintenance: https://primer.style/product/contribute/documentation/ and https://polaris-react.shopify.com/components

## Minimal Next Shape

- Create a small UI consistency spec for `canvas-workspace` that defines semantic token categories, shared component readiness, accessibility state coverage, and promotion rules; this follows the token/component/pattern split seen in Atlassian and Primer: https://atlassian.design/design-system and https://primer.style/product/
- Start with two or three repeated internal patterns, not a broad catalog: dock/tab controls, icon buttons/tool buttons, and loading/empty/error states are likely candidates because Storybook and Polaris both organize reuse around repeatable states and task patterns: https://storybook.js.org/docs/get-started/why-storybook and https://polaris-react.shopify.com/patterns
- Add examples or harness fixtures for state coverage before adding automated visual tooling; Storybook's value comes from reusable stories, and Primer asks for at least one Storybook story before broader sharing: https://storybook.js.org/docs/get-started/why-storybook and https://primer.style/product/contribute/adding-new-components/
